# ストレージ

## ディレクトリ構造

Calix はプロジェクトルート直下に `.calix/` ディレクトリを作成し、全てのバージョン管理データをここに格納する。

```
project_root/
└── .calix/
    ├── state.msgpack                          グローバル状態
    └── submodules/
        └── {submodule_id}/
            ├── info.msgpack                   サブモジュール情報
            ├── branches.msgpack               ブランチ一覧
            ├── merge_state.msgpack            (マージ中のみ)
            ├── rebase_state.msgpack           (リベース中のみ)
            └── commits/
                ├── {commit_id_1}.msgpack
                ├── {commit_id_2}.msgpack
                └── ...
```

### 各ファイルの役割

| ファイル | 内容 | 存在条件 |
|---|---|---|
| `state.msgpack` | グローバルタイムライン、サブモジュール一覧、シーケンス番号 | 常に存在 |
| `info.msgpack` | サブモジュールの種別、相対パス、現在のブランチ、HEAD | 常に存在 |
| `branches.msgpack` | 全ブランチの一覧と各ブランチの先端コミット ID | 常に存在 |
| `merge_state.msgpack` | マージ中のコンフリクト解決待ち状態 | マージ中のみ |
| `rebase_state.msgpack` | リベース中断中の進捗状態 | リベース中のみ |
| `{commit_id}.msgpack` | 個々のコミットデータ（差分、メッセージ、タイムスタンプなど） | コミットごとに1ファイル |

---

## シリアライズ形式

全てのデータは **MessagePack** 形式で保存される。JSON に比べてコンパクトでパース速度が速い。人間が直接読む必要がないバージョン管理データに適した選択である。

デバッグ時には `Commit::to_debug_json()` で JSON 出力を得ることができる。

---

## アトミック書き込み

ファイルの書き込みは全て以下のパターンで行われる。

1. 対象ファイルと同じディレクトリに `.tmp` 拡張子の一時ファイルとして書き込む
2. 書き込み完了後、`rename` で本来のファイルパスに移動する

```
書き込み対象: state.msgpack

  ① state.tmp に内容を書き出す
  ② state.tmp を state.msgpack にリネーム（アトミック操作）
```

これにより、書き込み途中でプロセスがクラッシュしても、ファイルが壊れた中途半端な状態になることを防ぐ。`rename` は POSIX ファイルシステムにおいてアトミックであることが保証されている。

全てのファイル書き込みは `storage::write_msgpack` 関数を経由する。この関数を使う限り、アトミック性は自動的に保証される。

---

## パスの扱い

### 相対パスの原則

`SubmoduleInfo.relative_path` はプロジェクトルートからの相対パスとして保存される。絶対パスはシリアライズされない。

これにより、プロジェクトディレクトリを別の場所に移動しても `.calix/` 内のデータは有効なまま保たれる。

### ストレージパスの構築

ストレージ内のファイルパスは `Repository` が一元管理する。サブモジュール ID からストレージルートへの変換は `Repository::submodule_storage_root()` が担当し、各モジュールが独自にパスを組み立てることはない。

---

## 一時ファイル

マージ中やリベース中の一時状態ファイル（`merge_state.msgpack`, `rebase_state.msgpack`）は、操作完了時に削除される。

- マージ: `resolve_conflict` 成功時に削除
- リベース: 全コミット適用完了時または `rebase_abort` 時に削除

これらのファイルの存在確認により、「現在マージ中か」「現在リベース中か」を判定できる。

---

## 設計上の制約

- 全てのファイル書き込みは `storage::write_msgpack` を経由すること。`std::fs::write` の直接使用は禁止
- コミットファイルは書き込み後に更新されない（不変）
- `info.msgpack` と `branches.msgpack` は状態変更のたびに全体を上書きする（差分更新ではない）
- バイナリファイル（動画・音声など）の読み書きは一切行わない
