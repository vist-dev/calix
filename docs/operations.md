# ブランチ操作

## ブランチの作成

分岐元のコミット ID を指定して新しいブランチを作る。同名のブランチが既に存在する場合はエラーになる。指定したコミットが存在しない場合もエラーになる。

作成直後のブランチは、分岐元コミットと同じ地点を指す。

## ブランチの切り替え

指定したブランチに HEAD を移動させる。存在しないブランチ名を指定するとエラーになる。

Calix はメタ情報のみを管理するため、切り替え時に「ファイルの中身を書き換える」といった処理は発生しない。内部のポインタが変わるだけである。

## ブランチの削除

- `main` ブランチは削除できない
- 現在チェックアウト中のブランチは削除できない

これらの制約は、履歴が失われる操作を防ぐためのガードレールである。

## マージ

2 つのブランチの変更を統合する操作。現在のブランチに、指定したブランチの変更を取り込む。

### 処理の流れ

1. 現在のブランチとマージ対象ブランチの共通祖先を探す
2. 共通祖先・現在の先端・対象の先端、それぞれの時点で状態を再構築する
3. 三方向マージで変更を統合する

### 三方向マージ

あるキーについて、共通祖先の状態を基準に「現在のブランチでの変更」と「対象ブランチでの変更」を比較する。

| 現在のブランチ | 対象ブランチ | 結果 |
|---|---|---|
| 変更なし | 変更あり | 対象の変更を採用 |
| 変更あり | 変更なし | 現在の変更を維持 |
| 同じ変更 | 同じ変更 | どちらかを採用（同一なので） |
| 異なる変更 | 異なる変更 | **コンフリクト** |

「変更」には値の更新だけでなく、キーの追加・削除も含まれる。一方が削除し、他方が更新した場合もコンフリクトになる。

### Fast-Forward

共通祖先が現在のブランチの HEAD と一致する場合、現在のブランチには独自の変更がない。この場合はマージコミットを作らず、ブランチポインタを対象ブランチの先端に移動するだけで済む。

```
[C1] ──→ [C2] ──→ [C3]  (feature)
          ↑
         main (HEAD)

  ↓ Fast-Forward

[C1] ──→ [C2] ──→ [C3]  (feature, main)
```

### コンフリクト発生時

コンフリクトが検出されると、マージは中断される。

- マージの状態（どのブランチ間で、共通祖先は何か）がストレージに保存される
- コンフリクトの詳細がエラーとして返される（どのキーで、どちらの値か）
- 呼び出し元は最終的な状態を決定し、`resolve_conflict` で解決を伝える
- 解決後にマージコミットが生成される

### マージコミット

通常のコミットとの違いは `second_parent_id` を持つこと。第一親は現在のブランチの前コミット、第二親はマージ対象ブランチの先端コミット。

状態再構築時は第一親のチェーンのみを辿る。マージコミットの差分には、マージによる変更が全て含まれているため、第二親側を辿る必要はない。

## リベース

現在のブランチのコミットを、別のブランチの先端に「付け替える」操作。履歴を直線的に整理するために使う。

### マージとリベースの違い

```
マージ後の履歴:
[C1] ── [C2] ── [M]  (main)
  └──── [F1] ───┘

リベース後の履歴:
[C1] ── [C2] ── [F1']  (feature)
```

マージは両方の履歴を保存して合流点を作る。リベースは一方の履歴を書き換えて直線にする。

### 処理の流れ

1. 現在のブランチとリベース先の共通祖先を探す
2. 共通祖先以降の現在ブランチのコミットを収集する
3. リベース先の HEAD を新しい基点として、各コミットの差分を順番に再適用する
4. 各コミットの適用時に三方向マージを行い、コンフリクトがないか確認する

### コミット ID の変更

リベースは新しいコミットを生成する操作である。元のコミットと同じ差分内容であっても、親コミットが異なるため新しい UUID が割り当てられる。リベース前のコミット ID はリベース後に無効になる。

### コンフリクト発生時

コミットの再適用中にコンフリクトが検出されると、リベースは中断される。

- リベースの状態（残りのコミット、現在の進捗など）がストレージに保存される
- コンフリクトの詳細がエラーとして返される
- 呼び出し元には 2 つの選択肢がある:
  - **続行**: 解決済みの状態を渡して `rebase_continue` を呼ぶ。残りのコミットの再適用が再開される
  - **中断**: `rebase_abort` を呼ぶ。ブランチの状態がリベース開始前に完全に復元される

### 安全性

リベースは破壊的操作であるため、開始前の HEAD コミット ID を必ず保存してから処理を開始する。`rebase_abort` はこの保存された ID を使ってブランチポインタを元に戻す。

## 共通祖先の探索

マージとリベースの両方で使われる基盤的な操作。

2 つのコミットから親を辿り、最初に合流する地点を共通祖先として返す。マージコミットによる DAG（有向非巡回グラフ）構造を考慮し、第二親も含めた幅優先探索で祖先を収集する。

共通祖先が見つからない場合（完全に独立した履歴）はエラーになる。
