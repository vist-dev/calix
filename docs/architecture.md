# アーキテクチャ

## Calix が解決する問題

動画編集ソフトでは、ユーザーが行った編集操作を「やり直し」「巻き戻し」「分岐して試す」といった形で管理したい。従来の線形 Undo/Redo では、途中の操作を捨てて分岐することや、複数の編集方針を並行して試すことが難しい。

Calix はこの問題を、git に似たコミット・ブランチ・マージのモデルで解決する。ただし git とは異なり、**管理対象はテキストファイルではなく、編集操作のメタ情報（キーと値のペア）**である。

## 何を管理し、何を管理しないか

### 管理するもの

- 編集パラメータの変更履歴（例: エフェクトの強度が 0.5 → 0.8 に変わった）
- パラメータの追加・削除の履歴
- 上記の変更がいつ、どの順番で、どの編集要素に対して行われたか

### 管理しないもの

- 動画・音声・画像などのバイナリファイル（コピーも読み取りもしない）
- ファイルパスの文字列は保持するが、そのファイルの中身には一切関与しない
- レンダリング結果やプレビューキャッシュ

この境界線により、Calix は軽量かつ高速に動作する。バージョン管理のデータサイズはバイナリアセットに依存しない。

## レイヤー構造

```
┌─────────────────────────────────────────────┐
│  呼び出し元（C++ 動画編集ソフト / FFI）       │
└─────────────┬───────────────────────────────┘
              │
┌─────────────▼───────────────────────────────┐
│  公開 API 層（lib.rs）                        │
│  Repository / Submodule / Commit / Diff      │
└─────────────┬───────────────────────────────┘
              │
┌─────────────▼───────────────────────────────┐
│  ドメインロジック層                            │
│  ブランチ管理 / マージ / リベース / 差分計算    │
│  (branch.rs, submodule.rs, diff.rs)          │
└─────────────┬───────────────────────────────┘
              │
┌─────────────▼───────────────────────────────┐
│  永続化層（storage.rs）                       │
│  MessagePack シリアライズ / アトミック書き込み   │
└─────────────┬───────────────────────────────┘
              │
        ファイルシステム
```

各層の責務は明確に分かれている。

- **公開 API 層** — 外部に公開する型と操作の入口。内部モジュールの詳細を隠蔽する
- **ドメインロジック層** — バージョン管理の中核ロジック。ブランチ分岐、三方向マージ、コンフリクト検出など
- **永続化層** — ファイル I/O を一箇所に集約。全ての書き込みは `storage::write_msgpack` を経由し、アトミック性を保証する

## データフロー

### コミット作成の流れ

1. 呼び出し元が「変更前の状態」と「変更後の状態」を渡す
2. `Diff::from_states()` が差分を計算する
3. 差分を持つ `Commit` を構築する
4. `Submodule::append_commit()` がコミットファイルを書き込み、ブランチの先端を更新する

### 状態復元の流れ

1. 指定されたコミット ID から `parent_id` チェーンを辿り、ルートまで遡る
2. ルートから順に各コミットの差分を適用する
3. 最終的な `HashMap<String, String>` が、そのコミット時点の完全な状態となる

状態は常にオンデマンドで計算される。スナップショットを保存しないため、ストレージ効率が高い。

## git との設計上の違い

| 観点 | git | Calix |
|---|---|---|
| 管理単位 | リポジトリ全体で1つのブランチ | サブモジュールごとに独立したブランチ |
| 管理対象 | テキスト / バイナリファイルの内容 | キー・値ペアのメタ情報のみ |
| ブランチの実体 | refs/heads 配下のファイル | BranchStore 内の HashMap エントリ |
| 差分の粒度 | 行単位のテキスト差分 | キー単位の set/remove |
| ワーキングツリー | ファイルシステムに展開 | なし（状態はオンデマンド計算） |
| マージ戦略 | 再帰的三方向マージ + 行レベル | 三方向マージ（キーレベル） |

## モジュール構成

```
src/
├── lib.rs          公開 API の入口
├── error.rs        エラー型とコンフリクト情報
├── commit.rs       Commit / Diff / GlobalOrder
├── branch.rs       Branch / BranchStore
├── submodule.rs    Submodule とブランチ・マージ・リベース操作
├── global.rs       Repository / グローバルタイムライン
├── storage.rs      MessagePack I/O
└── diff.rs         三方向マージ
```

## 将来の拡張ポイント

- **C++ FFI レイヤー** — `lib.rs` の公開 API を `extern "C"` 関数として公開する。`crate-type` に `cdylib` / `staticlib` を含んでいるのはこのため
- **状態スナップショットキャッシュ** — コミットチェーンが長くなった場合の再構築コスト軽減。一定間隔でスナップショットを保存し、そこからの差分適用で復元する
- **proptest によるプロパティテスト** — 差分のラウンドトリップ（apply → from_states → apply が元の状態と一致する）や、マージの交換法則の検証
